WHAT THIS FILE IS
-----------------
You (the next Manus agent, or the human developer who picks this up) must
re-deploy this exact repository as a NATIVE ANDROID APP — NOT as a web app.
The codebase is an Expo / React Native project with a small Node.js (tRPC)
backend. Web is supported only as a development preview; the production
artifact is an Android APK / AAB.

Read every section before touching anything. Do NOT make code changes
unless explicitly required. Speed of delivery is less important than
correctness — a single bad migration or unsigned APK can lock all
landlords out of the system overnight.

This rev (3) is **autonomous-friendly**: every previous "ask the user"
gap has been replaced with a concrete file or command. The only thing
the agent still needs from a human is the values listed in
DEPLOY_PREREQUISITES.md (secrets + keystore). After that, every step
below is mechanical.

1. HARD CONSTRAINTS (do not violate)
================================================================================

  * NO web-app deployment. The mobile app is the deliverable. The web
    bundle is dev-only and must NOT be published as the primary product.

  * NO Manus OAuth in production. Login is EMAIL + PASSWORD only.
    The /login screen on this codebase already supports username+password
    via the `auth.login` tRPC mutation. The OAuth callback route still
    exists in source but MUST NOT be reachable in the deployed app.
    Specifically:
        - Do not set `OWNER_OPEN_ID` in env.
        - Do not set `OAUTH_SERVER_URL` in env.
        - Do not advertise an OAuth button anywhere in the UI.
    `pnpm verify:deploy` enforces this.

  * NO leftover seed accounts. The bundled `pnpm seed:admin` script wipes
    the users table (and dependent rows) before seeding the single admin.
    No demo landlord, no demo tenant, no test admin. Only the admin
    configured in DEPLOY_PREREQUISITES.md.

  * ALWAYS deploy the latest tagged version. Before building, run:
        git fetch --tags
        git tag --sort=-v:refname | head -5
    Use the topmost tag (e.g. v1.6.0). If no tags exist, run:
        git tag v$(node -p "require('./constants/app-version.ts')") -m "auto-tag"
        git push --tags
    Bump constants/app-version.ts AND app.config.ts in lock-step if you
    intentionally cut a new release. NEVER ship an older version once a
    newer one exists.

  * The schema MUST be REBUILT FROM SCRATCH on each new deployment until
    section 5's verification check passes. This guarantees JDK-21 / MySQL-8
    compatibility and removes drift from old environments.

  * Do NOT modify code unless a step in this document explicitly says so.
    The current build is the locked release.

  * Do NOT downgrade any pinned versions. If a tool insists on it, stop
    and report — there's almost certainly a better fix.

  * Do NOT edit files inside `android/` or `ios/` if they exist. Native
    project files are regenerated by Expo prebuild. Edit `app.config.ts`
    + `expo-build-properties` instead.

  * Do NOT commit a `.env`, the SQL dump, the seeded admin password, or
    any APK file to the repo. The seed password in DEPLOY_PREREQUISITES.md
    is bootstrap-only and must be rotated immediately after first sign-in.

================================================================================
4a. AUTO-LOADING SECRETS FROM .secrets.local.txt (sections 3a)
================================================================================

The project root MAY contain a file named `.secrets.local.txt` (gitignored).
It looks like this:

    # comments allowed
    EXPO_TOKEN=expo_abc123...
    BOOTSTRAP_ADMIN_EMAIL=jarren.manusai@outlook.com
    BOOTSTRAP_ADMIN_PASSWORD=...
    # JWT_SECRET=optional

WHEN THIS FILE EXISTS, THE AGENT MUST:

  1. Parse it line-by-line. Skip blank lines and lines starting with `#`.
  2. For each `KEY=VALUE` line, call `webdev_request_secrets` with that
     KEY/VALUE pair so the value is registered in the deploy environment.
  3. NEVER print the VALUE in chat or logs. Print only the KEY name.
  4. NEVER copy `.secrets.local.txt` into a checkpoint or commit it. It
     is in `.gitignore` and must stay there.
  5. After loading, the agent only asks the human about secrets NOT
     present in the file. Specifically:
       - DATABASE_URL is auto-provisioned by Manus webdev runtime.
       - EXPO_PUBLIC_API_URL is set by the agent itself AFTER server
         deploy (see Phase 2F).
       - Anything else missing: ONE blocking question.
  6. Validate the file's content:
       * Reject any line containing `OAUTH_SERVER_URL` or `OWNER_OPEN_ID`
         (production must use email+password only, hard rule §1).
       * Warn if `BOOTSTRAP_ADMIN_PASSWORD` looks too short (<10 chars)
         or matches a known weak default.
  7. After all secrets are registered, run `pnpm verify:deploy`. The
     auditor checks that EXPO_TOKEN, JWT_SECRET, and DATABASE_URL are
     present and that no OAuth env vars are set.

A template lives at `.secrets.template.txt` (committed). Operators copy
it, fill values, and drop the result into the sandbox before starting
the Manus session.

IF NO .secrets.local.txt EXISTS:
  Fall back to the standard §4 flow: ask the operator for each secret
  individually via `webdev_request_secrets` cards.


================================================================================
4b. OVERRIDING MANUS RUNTIME-INJECTED SECRETS
================================================================================

The Manus webdev sandbox auto-injects three values whose runtime
defaults VIOLATE this project's deploy contract:

  RUNTIME-INJECTED          DEPLOY CONTRACT REQUIRES
  ------------------------- ----------------------------------------------
  OAUTH_SERVER_URL          empty/unset (§1 hard rule)
  OWNER_OPEN_ID             empty/unset (§1 hard rule)
  JWT_SECRET (22 chars)     ≥32 chars (§4 + verify:deploy enforces)

The agent MUST override all three using `webdev_request_secrets` with
`preventMatching: true`. This is a documented, supported override path
— NOT a hack. Do not invent runtime workarounds (do not edit
scripts/load-env.js, do not `delete process.env.X` at boot).

EXACT TOOL CALL (paste-ready)
-----------------------------

  // Step 1: disable OAuth (both vars must be empty strings)
  webdev_request_secrets({
    secrets: [
      { key: "OAUTH_SERVER_URL", value: "", preventMatching: true,
        description: "MUST be empty per MANUS_HANDOFF.txt §1 — disables Manus OAuth" },
      { key: "OWNER_OPEN_ID",    value: "", preventMatching: true,
        description: "MUST be empty per MANUS_HANDOFF.txt §1 — disables Manus OAuth" }
    ]
  })

  // Step 2: install a strong JWT_SECRET (do NOT print the value to chat)
  //   Generate with:
  //     node -e "console.log(require('crypto').randomBytes(48).toString('hex'))"
  webdev_request_secrets({
    secrets: [{
      key: "JWT_SECRET",
      value: "<96-char random hex>",
      preventMatching: true,
      description: "Production JWT signing secret (overrides the 22-char built-in)"
    }]
  })

  // Step 3: restart the dev server so the new env values are loaded
  webdev_restart_server({ brief: "reload after secret override" })

AFTER OVERRIDING, RE-RUN: `pnpm verify:deploy`. All three previously
failing audit lines should now pass:

  ✓  DATABASE_URL set
  ✓  JWT_SECRET set & strong
  ✓  OAuth disabled (no OAUTH_SERVER_URL / OWNER_OPEN_ID)

WHY THIS WORKS
--------------

  * The OAuth gate in `server/_core/index.ts` is
      `if (process.env.OAUTH_SERVER_URL && process.env.OWNER_OPEN_ID)`.
    Empty strings are falsy in JavaScript, so the gate stays closed and
    `/api/oauth/*` routes never mount.
  * `verify:deploy` checks the same falsy semantics for the OAuth pair
    and a length-≥32 check for JWT_SECRET. Empty strings + a 96-char
    secret both satisfy the audit.
  * `preventMatching: true` tells webdev_request_secrets NOT to fall
    back to the BYOK auto-match (which would re-inject the runtime
    value).

DO NOT confuse "override" with "delete". The values still exist (as
empty strings); they're just truthy-falsy in the right way.

5. DATABASE: REBUILD FROM SCRATCH (every new deployment)
================================================================================

The schema lives in `drizzle/schema.ts`. Migrations that must all be
applied IN ORDER on a fresh database:

  drizzle/0000_elite_eternals.sql           (initial bootstrap)
  drizzle/0001_military_typhoid_mary.sql    (multi-tenant + roles + APK releases + reset tokens etc.)
  drizzle/0002_sharp_quentin_quire.sql      (adds `users.phone` for PH validation)

  (If newer migrations exist when you read this — drizzle/0003_*.sql or
   higher — apply them in numeric order too. NEVER skip; never reorder.)

NEW DEPLOYMENT — full rebuild procedure
----------------------------------------

Always do this on every brand-new environment. Skipping it leads to schema
drift between dev and prod and JDK-21-incompatible binary protocol issues
on legacy MySQL clients.

  Step 1. Provision a fresh MySQL 8 instance. Confirm:
              SELECT VERSION();          -- must be 8.0+
              SHOW VARIABLES LIKE 'character_set_database';
              -- must be utf8mb4
              SHOW VARIABLES LIKE 'collation_database';
              -- must be utf8mb4_unicode_ci or utf8mb4_0900_ai_ci

  Step 2. Drop and recreate the database to guarantee a clean slate:
              DROP DATABASE IF EXISTS utilitybill;
              CREATE DATABASE utilitybill
                  CHARACTER SET utf8mb4
                  COLLATE utf8mb4_unicode_ci;

  Step 3. Apply migrations IN ORDER. Either:

          Option A — recommended (one shot):
              pnpm db:push
              # which is: drizzle-kit generate && drizzle-kit migrate

          Option B — manual:
              mysql -u <user> -p utilitybill < drizzle/0000_elite_eternals.sql
              mysql -u <user> -p utilitybill < drizzle/0001_military_typhoid_mary.sql
              mysql -u <user> -p utilitybill < drizzle/0002_sharp_quentin_quire.sql
              # repeat for any drizzle/0003_*.sql, etc.

  Step 4. Verify by running `pnpm verify:deploy`. It checks tables,
          columns, the admin row, and version consistency in one shot.

  Step 5. Backup discipline:
          Take a `mysqldump --single-transaction --routines` snapshot before
          ANY future schema migration in production.

6. SEED EXACTLY ONE ADMIN — `pnpm seed:admin`
================================================================================

The repo ships `scripts/seed-admin.ts` (called by `pnpm seed:admin`)
which:

  1. Refuses to run unless DATABASE_URL is set, and refuses on
     NODE_ENV=production unless ALLOW_PROD_SEED=1.
  2. Toggles FOREIGN_KEY_CHECKS off, deletes every row from every table
     in the right child-first order, toggles them back on.
  3. Hashes the admin password using the project's real auth helper
     (`server/auth.ts → hashPassword`, bcrypt cost 10 — same as runtime).
  4. Inserts the admin via `db.createUser` (the same code path used by
     `auth.register`).
  5. Verifies the row landed and prints a summary.

  Defaults (override via env if needed):
      ADMIN_EMAIL    = jarren.manusai@outlook.com
      ADMIN_PASSWORD = 030921manusai!@!
      ADMIN_NAME     = Jarren (Admin)
      ADMIN_PHONE    = +639000000000

To run:

      pnpm seed:admin

To override a value (one-shot):

      ADMIN_EMAIL=other@example.com ADMIN_PASSWORD='different' pnpm seed:admin

POST-SEED HARDENING (mandatory)
-------------------------------

  1. Sign in to the deployed app once with the admin email + password to
     confirm the credentials work.
  2. Inside the app, immediately rotate the password from
     Settings → Security to a value of the operator's choosing. The
     default password in this document is bootstrap-only.
  3. Inform the operator (the human who owns the configured email
     address) that the admin account is now live and that they should
     change the password on first sign-in.
  4. Delete or scrub the seed-script log lines from the deploy logs so
     the bootstrap password isn't preserved in history.

EAS Build (cloud, no local Android SDK needed) — the supported path
--------------------------------------------------------------------

The repo includes a pre-configured `eas.json` with profiles pinned to
the JDK 21 image (`ubuntu-22.04-jdk-21-ndk-r27b`). You do NOT need to
run `eas build:configure`.

  1) Install EAS CLI ad-hoc (or globally if you prefer):
       npx eas-cli --version
       # If missing globally: npm i -g eas-cli@latest

  2) Authenticate non-interactively. The agent uses an `EXPO_TOKEN`
     supplied by the operator (DEPLOY_PREREQUISITES.md §5):
       export EXPO_TOKEN=<token>
       npx eas-cli whoami    # confirms login

  3) Configure credentials. The first build will prompt you to
     generate or upload a keystore. Per DEPLOY_PREREQUISITES.md §4
     the operator already chose option A / B / C. Apply that:
       - Option A (new EAS-managed): just run the build; EAS creates one.
       - Option B (existing EAS-managed): nothing to do; reused automatically.
       - Option C (BYO keystore): import once with
           npx eas-cli credentials  → Android → production → Set up a new
         keystore → "Upload a keystore" → paste the four secrets.

  4) Build (NON-INTERACTIVE — required for hands-off agent runs):
       export EAS_NO_VCS=1                # don't fail on uncommitted changes
       npx eas-cli build --platform android --profile production \
         --non-interactive \
         --no-wait                        # returns immediately with a build URL
     Wait for the cloud build to finish (~10–25 min). The dashboard
     produces a downloadable .apk URL. Without --non-interactive EAS
     will block on a TTY prompt for keystore generation, project link,
     etc. — which a sandbox agent cannot answer.

     For Option A (brand-new EAS-managed keystore) the FIRST build still
     needs to authorise keystore creation. With --non-interactive +
     EXPO_TOKEN this is auto-approved silently — BUT only if the project
     already has an `extra.eas.projectId` (see AGENT_QUICKSTART.md
     "First-time-ever EAS bootstrap" — do `eas init` LOCALLY first).

  5) (Optional) Build an .aab for Play Store:
       EAS_NO_VCS=1 npx eas-cli build --platform android --profile production-aab \
         --non-interactive --no-wait

Distribution
-------------

FIRST APK (no previous version installed on user phones):
  Use whichever channel DEPLOY_PREREQUISITES.md §6 specifies. Common
  choices:
    - Direct link: upload .apk to the Manus storage proxy or any HTTPS

    - Play Console internal testing: `eas submit -p android`.

SUBSEQUENT UPDATES (in-app self-updater):
  After the first install, every future update goes through:

    1) Sign in as admin.
    2) Open Settings → "App updates".
    3) Use the dashed dropzone to upload the new APK file.
    4) Type the new version (must match constants/app-version.ts).
    5) Tap "Deploy".

  install on Android.

  NOTE: All future APKs MUST be signed with the same keystore as the
  first one. Otherwise users see "App not installed".

11. TROUBLESHOOTING (the 90% of issues you'll actually hit)
================================================================================

PROBLEM: "[Database] Cannot upsert user: database not available" on startup
  → DATABASE_URL is wrong or unreachable. Check SSL flag and host:port.

PROBLEM: /login accepts the admin password but lands on a blank screen
  → The role on the row isn't 'admin'. Re-run `pnpm seed:admin`.

PROBLEM: Admin can't see the App Updates panel
  → users.role in DB is something other than 'admin'. Re-seed.

PROBLEM: Registration screen shows "Verification challenge invalid"
  → JWT_SECRET changed between issue and verify. Make sure both the auth
    cookies AND the captcha module read the same secret. They do by
    default. Restart the server after rotating the secret.

PROBLEM: "Property 'phone' is missing" TypeScript error in tests
  → You ran the project against a DB that doesn't have the 0002
    migration applied. Apply it (section 5).

PROBLEM: APK install fails with "App not installed" on the device
  → Either: (a) the user has an older signed copy with a different
    keystore — they must uninstall first, or you must sign with the same
    keystore as before; (b) the device has < Android 7 (minSdk 24).

PROBLEM: EAS Build fails with "Could not find tools.jar"
  → JRE installed instead of JDK. Install full JDK 21 (Temurin).

PROBLEM: Gradle says "Unsupported class file major version 65"
  → JDK too NEW for the locked Gradle. Use JDK 21, NOT JDK 22+.

PROBLEM: "react-native-reanimated" runtime crash
  → babel.config.js missing the reanimated plugin. Don't edit this file
    unless absolutely necessary; the template version is correct.

PROBLEM: Web preview works but native build fails
  → Almost always a missing `expo-build-properties` config. Check
    app.config.ts plugins array.

PROBLEM: User reports "I can't log in with Google"
  → Working as designed. OAuth has been intentionally disabled. Tell the
    user to use email + password.

PROBLEM: Password reset link never arrives by email
  → The current code path logs the reset URL to the server console
    instead of sending email. Email delivery is intentionally a TODO —
    users either copy the URL from the operator or use the in-app
    "rotate password" flow once signed in.
